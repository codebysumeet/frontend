# Krishi Clinic Lite

![CI](https://github.com/YOUR_GITHUB_USERNAME/krishi-clinic-lite/actions/workflows/ci.yml/badge.svg)

A crop disease advisory dashboard. A user uploads a photo of a crop, picks the crop type, and optionally adds field notes. An AI provider analyzes the image and returns a disease diagnosis, confidence score, severity, and a treatment recommendation. Every prediction is stored in Postgres, searchable in a history view, and rolled up into an analytics dashboard.

Built for the GramIQ Full Stack Developer Intern technical assignment ("Krishi Clinic Lite").

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Frontend | Next.js 14 (App Router) + TypeScript + Tailwind | Required stack. App Router keeps routing simple for 4 flat views; Tailwind keeps styling fast without a design system to maintain. |
| Charts | Recharts | Lightweight, composable, plays well with React state/props directly. |
| Backend | FastAPI + Pydantic | Required stack. Async-friendly, auto-generates OpenAPI docs, Pydantic gives free request validation. |
| ORM / Migrations | SQLAlchemy (sync) + Alembic | Required stack. Sync SQLAlchemy chosen over async for simplicity at this scope — see `ENGINEERING_DECISIONS.md`. |
| Database | PostgreSQL 16 | Required stack. |
| AI Integration | Mock provider (default) | Deterministic, no API key needed, graded identically to a real provider per the brief. Architecture supports adding Gemini/OpenAI with a one-line config change — see below. |
| DevOps | Docker Compose + GitHub Actions | Required stack. One command brings up the full stack; CI lints and tests both services on every push/PR to `main`. |

## Setup instructions

### Prerequisites
- Docker and Docker Compose installed
- That's it — no local Node, Python, or Postgres installation required.

### Run everything with one command

```bash
git clone https://github.com/YOUR_GITHUB_USERNAME/krishi-clinic-lite.git
cd krishi-clinic-lite
docker compose up --build
```

This brings up three containers:
- `db` — Postgres 16
- `backend` — FastAPI on `http://localhost:8000` (runs Alembic migrations and seeds ~24 sample predictions automatically on first boot)
- `frontend` — Next.js on `http://localhost:3000`

Open **http://localhost:3000** — the History and Analytics views will already have sample data; the Upload view is ready to accept a real image immediately.

API docs (auto-generated by FastAPI) are available at **http://localhost:8000/docs**.

### Running services individually (optional, for local development)

**Backend:**
```bash
cd backend
python -m venv venv && source venv/bin/activate
pip install -r requirements-dev.txt
cp .env.example .env   # point DATABASE_URL at a local Postgres if not using Docker
alembic upgrade head
python seed.py
uvicorn app.main:app --reload
```

**Frontend:**
```bash
cd frontend
npm install
cp .env.example .env.local
npm run dev
```

**Running tests:**
```bash
cd backend
pytest -v
```

## Architecture overview

```
┌────────────┐        ┌──────────────────────────────────────────────┐
│  Frontend  │  HTTP  │                    Backend                   │
│  Next.js   │───────▶│  Routes (thin) → Services → SQLAlchemy → DB  │
│            │        │                     │                        │
│ services/  │        │                     ▼                        │
│  api.ts    │        │              AIProvider (interface)           │
│ (1 file,   │        │             /            \                   │
│  all calls)│        │      MockProvider    GeminiProvider (stub)    │
└────────────┘        └──────────────────────────────────────────────┘
                                     │
                                     ▼
                              PostgreSQL (predictions table)
```

- **Routes** (`app/api/routes/*.py`) only parse the request and call a service method. No business logic lives here.
- **Services** (`app/services/prediction_service.py`) hold all business logic: validation, calling the AI provider, persisting records, computing analytics.
- **AI Provider** (`app/services/ai/*.py`) is an abstract interface (`AIProvider.analyze()`). The service layer never knows whether it's talking to a mock, Gemini, or a local model — see "AI integration" below.
- **Storage** (`app/storage/local.py`) is similarly abstracted behind an `ImageStorage` interface, so swapping local disk for S3/Azure Blob later touches one file.
- **Frontend** never calls `fetch` directly from a component — everything routes through `services/api.ts`, which also normalizes error handling into one `ApiError` type.

## AI integration

The default and only wired-up provider is a **deterministic mock** (`app/services/ai/mock_provider.py`). It has no external dependency and no API key — same behavior every time for a given image, which also makes it trivially testable.

The abstraction:

```python
class AIProvider(ABC):
    @abstractmethod
    async def analyze(self, image_bytes, crop_type, farmer_notes=None) -> PredictionResult:
        ...
```

Routes and services depend only on this interface via `app/services/ai/factory.py`, which reads the `AI_PROVIDER` env var (`mock` by default) and returns the matching implementation. Adding Gemini (or any other real provider) means:

1. Writing one new class implementing `analyze()` (e.g. `app/services/ai/gemini_provider.py`).
2. Adding one branch to the factory.
3. Setting `AI_PROVIDER=gemini` and `GEMINI_API_KEY=...` in the environment.

Nothing in routes, models, or the frontend changes.

## Image storage

Images are stored on local disk (`backend/uploads/`, mounted as a Docker volume so they survive container restarts) and served back at `/uploads/{filename}`. This is behind an `ImageStorage` interface (`app/storage/local.py`) with `save()` and `url_for()` methods — swapping in S3 or Azure Blob later means implementing one new class, not touching the service or routes.

## Known limitations / what I'd improve with more time

- **No image preview persistence across the Upload → Detail flow beyond the single session** — the Upload Panel shows a preview from the local file object; the stored image on disk isn't yet rendered back in the History/Detail views (the URL is available via the API, just not wired into the `<PredictionDetail>` component's UI).
- **No authentication** — this is a single-tenant internal tool per the brief's scope; a real deployment would need at least basic auth in front of the dashboard.
- **AI provider is mock-only** — the abstraction supports a real provider, but only the mock is implemented and tested end-to-end, per the assignment's guidance that provider choice/accuracy isn't graded.
- **No retry/backoff on AI provider failures** — a real provider call is wrapped in a try/except that surfaces a clean 502, but doesn't retry. Worth adding once a real (network-dependent) provider is in place.
- **Pagination is offset-based**, not cursor-based — fine at this scale, would reconsider for very large tables.
